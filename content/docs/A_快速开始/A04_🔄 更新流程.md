# 🔄 更新流程

本文档定义了设计规范的更新规则与协作流程,确保所有变更都是可控、可追溯、可协同的。

---

## 更新原则

### 向后兼容优先

- 尽量通过扩展而非破坏现有规则
- 必须破坏性更新时,应提供明确的迁移方案
- 当需要用新组件替换就组件时建议保留 2 个版本周期

### 小步快走

- 优先小范围验证,再推广到全局
- 避免一次性大规模重构

### 记录与透明

- 所有更新必须同步到 [[A06_📝 更新日志|更新日志]]
- 重要变更需要通过协作群同步告知
- 文档中的 `last_updated` 属性必须更新，记录最后一次的更新日期

---

## 规范&组件状态管理

所有文档及组件使用 **status** 属性标记当前状态：

| 状态          | 含义          | 适用场景          |
| ----------- | ----------- | ------------- |
| `Draft`     | 草稿，未对外发布    | 新建文档、尚未完成的内容  |
| `In_review` | 审阅中         | 需要团队评审的更新     |
| `Published` | 已发布，对外可见    | 正式生效的规范内容     |
| `Archived`  | 已归档，保留但不再维护 | 已废弃但需要保留的历史文档 |

> 提示：未填写 `status` 时视为 `Published`



## 更新类型与流程

### 1. 新增内容

**适用场景**：新增组件、新增规范章节、新增设计模式等

**流程**：
1. 创建新文档，设置 `status: Draft`
2. 完成初稿后，更新为 `status: In_review`
3. 在团队内部征求反馈（可使用 Figma 评论 / 协作群讨论）
4. 修订完成后，更新为 `status: Published`
5. 在 [[A06_📝 更新日志|更新日志]] 中记录新增内容
6. 同步到 Figma 组件库（如适用）
7. 在团队群公告周知

**检查清单**：
- [ ] 文档包含完整信息（status, last_updated）
- [ ] 内容符合 [[A02_📏 设计原则|设计原则]]
- [ ] 在 [[00_📚 内容索引|内容索引]] 中添加导航链接
- [ ] 更新日志已记录
- [ ] Figma 组件已同步（如适用）



### 2. 优化现有内容

**适用场景**：修改措辞、补充说明、优化案例等

**流程**：
1. 直接修改文档
2. 更新 `last_updated` 日期
3. 如果是重要调整，在 [[A06_📝 更新日志|更新日志]] 中记录

**检查清单**：
- [ ] `last_updated` 已更新
- [ ] 如有 Figma 引用，需同步检查
- [ ] 重要调整已记录到更新日志



### 3. 破坏性更新

**适用场景**：修改命名规则、删除组件、调整基础规范等会影响已有设计的更新

**流程**：
1. 在团队内发起讨论，确认更新必要性
2. 准备迁移方案（如：新旧对照表、替换指南）
3. 标记旧内容为 Deprecated（而非直接删除）
4. 创建新内容，设置 `status: In_review`
5. 团队评审通过后，发布新内容
6. 在 [[A06_📝 更新日志|更新日志]] 中**高亮说明**破坏性更新
7. 至少保留 2 个版本周期后，才可将 Deprecated 内容更新为 `status: Archived`

**检查清单**：
- [ ] 已在团队内讨论并达成共识
- [ ] 旧内容已标记 Deprecated 或添加醒目提示
- [ ] 迁移方案已明确（如：替换对照表、代码示例）
- [ ] 更新日志中已高亮说明
- [ ] 团队已充分告知

> **破坏性更新示例**
> - 删除某个颜色 Token
> - 修改组件 API
> - 调整间距体系数值
> - 改变命名规则



### 4. 归档内容

**适用场景**：已被替代的规范、不再适用的模式

**流程**：
1. 确认内容已不再使用或已有替代方案
2. 更新文档 `status: Archived`
3. 在文档顶部添加醒目提示，说明替代方案
4. 从 [[00_📚 内容索引|内容索引]] 的主导航中移除（但保留文档）
5. 在 [[A06_📝 更新日志|更新日志]] 中记录

**归档文档示例提示**：
```markdown
> 此文档已归档
> 本规范已被 [[新规范名称]] 替代，请使用新规范进行设计。
```


## 更新日志规范

每次更新需要在 [[A06_📝 更新日志|更新日志]] 中记录，格式如下：

```markdown
v版本号 · 日期
- **模块名称**：更新内容简述
- **模块名称**：破坏性更新 ⚠️
```

**示例**：
```markdown
v1.3.0 · 2026.02.15
- **颜色系统**：新增品牌色辅助色 `brand-secondary`
- **按钮组件**：新增 `ghost` 样式
- **间距规范**：调整基础单位从 4px 改为 8px ⚠️
```



## 常见问题

### Q: 什么时候需要发新版本？

**A**: 以下情况需要发布新版本：
- 新增或修改基础规范（颜色、字体、间距等）
- 新增或删除组件
- 破坏性更新
- 重要的优化调整

文案修订、案例补充等小改动无需发版本。



### Q: 如何处理紧急修复？

**A**: 紧急 Bug 修复可以跳过评审流程，但必须：
1. 立即修改并标记 `status: Published`
2. 在 24 小时内补充更新日志
3. 在团队群同步告知

---

### Q: 多人同时更新如何避免冲突？

**A**: 
- 更新前先拉取最新版本
- 大范围更新提前在群里告知
- 使用 Git 分支进行并行开发（如适用）
- Figma 组件库由设计负责人统一管理
